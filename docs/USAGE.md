# 小红书监控系统使用文档

一个针对小红书的内容监控 + AI 改写 + 数据同步工具，适合做内容运营、竞品分析、爆款追踪。

## 核心功能一览

| 模块 | 能做什么 |
|------|----------|
| **多账号管理** | 扫码登录小红书、cookie 健康检查、自动失效告警 |
| **帖子监控** | 监控自己/他人帖子的点赞、收藏、评论增量；阈值告警推送飞书/企微 |
| **热门内容抓取** | 按关键词定时搜索热门帖子、按点赞数过滤 |
| **AI 文案改写** | 接 OpenAI/DeepSeek 等 API，多 Prompt 模板，按需触发改写 |
| **飞书同步** | 一键把选中的热门帖子（含改写文案）批量写入多维表格 |
| **推送通知** | 增量告警、新评论、Cookie 失效、每日数据日报 |

## 快速开始

### 本地启动

```bash
# 1. 装依赖
uv sync --python 3.11
uv run playwright install chromium

# 2. 起服务（含前端）
uv run uvicorn api.main:app --port 8080
```

打开 http://localhost:8080 ，**默认账号 `admin / admin123`**。

### 服务器部署

服务器需要 Python 3.11+、约 1.5G 内存（Playwright 用）、阿里云 ICP 备案（如果走域名）或自签证书 + 高位端口（如果走 IP）。

详细部署步骤已经写过了——简单来说：rsync 项目代码 → `uv sync` → `playwright install chromium` → systemd 服务监听 8003。

## 使用流程（按顺序操作）

### Step 1 — 添加账号（必做）

任何抓取都需要小红书 cookie。

1. 进 **「监控设置」→ 账号管理**
2. 填账号名称（自己起的标识，比如 `小号A`）
3. 代理 URL **可以留空**。说明：
   - **代理只在监控帖子数据时用**，扫码登录始终用本机直连
   - 支持 `http://user:pass@host:port`
   - **不支持 `socks5://user:pass@...`**（Playwright 限制），建议用 HTTP 代理
4. 点「**扫码登录**」→ 用小红书 App 扫弹窗里的二维码
5. 登录成功后账号自动保存

### Step 2 — 添加监控帖子

进 **「帖子监控」→ 添加帖子**：

1. **分组**选择：
   - **我的帖子**：自己账号发布的内容
   - **观测帖子**：竞品/他人帖子（默认）
2. **链接**：每行粘一个，支持小红书 App 分享短链（`xhslink.com/...`）或完整 URL
3. **绑定账号**（可选）：
   - 我的帖子建议绑定对应的账号
   - 观测帖子默认不绑（保持匿名访问）

### Step 3 — 设置告警规则

进 **「监控设置」**：

- **检测间隔**：建议 30 分钟（最低 1 分钟）
- **告警规则**（点赞/收藏/评论）：
  - 启用开关
  - 阈值：单次检测期间增量超过这个值就推送
- **每日报告**：到点把当天数据汇总推送一次

### Step 4 — 配推送渠道

支持企业微信 + 飞书机器人，二选一或都配：

- **企业微信**：群机器人 webhook URL
- **飞书机器人**：群自定义机器人 webhook URL

### Step 5（可选）— 热门内容抓取

进 **「监控设置」→ 热门内容抓取**：

1. 启用开关
2. 关键词（中英文逗号分隔，如 `ai，claude，创业`）
3. 最低点赞数（默认 1000，会过滤低于此值的帖子）
4. 监控的账号：勾选 1-2 个有效账号（不勾默认用所有活跃账号）
5. **保存设置**（这一步常被忽略）

然后到 **「热门内容」**页面，点「**立即抓取**」开始第一次抓取。后续每 30 分钟（或检测间隔）自动跑一次。

### Step 6（可选）— AI 改写 + 同步飞书表格

**6.1 配 AI**：「监控设置 → AI 配置」填 base URL、API Key、模型名。支持任何 OpenAI 协议兼容的服务（DeepSeek、千问、本地 Ollama 等）。

**6.2 管理 Prompt 模板**：「监控设置 → 改写 Prompt 模板」。

- 默认有一个 prompt
- 可以新建多个（比如「爆款风」「干货风」「故事化」）
- 每个 prompt **必须包含 `{content}` 占位符**
- 设置默认 prompt（弹窗里默认选它）

**6.3 在「热门内容」页操作**：

- 点击行内的标题或 ✨ 图标 → **弹窗显示原文**
- 在弹窗里下拉选 Prompt 模板 → 点「**改写**」→ 几秒后看结果
- 同一帖子可换 Prompt 反复改写
- 满意后关闭弹窗

**6.4 配飞书多维表格**：「监控设置 → 飞书多维表格同步」

- 飞书自建应用：App ID + Secret
- 多维表格：从 URL 解析 App Token + Table ID

**关键的飞书权限要点（坑）**：

1. 应用必须开通**「应用身份」（不是用户身份）**的 `bitable:app` 权限
   - 路径：飞书开放平台 → 你的应用 → 权限管理 → 筛选「应用身份」搜 `bitable:app`
2. 应用必须被加为**目标多维表格的协作者**（可编辑权限）
   - 路径：打开多维表格 → 右上「···」分享 → 添加协作者 → 搜应用名称
3. 字段会**自动创建**（关键词/标题/原文/改写文案/点赞数/收藏数/帖子链接/作者），不需要手动建表头

**6.5 同步到飞书**：

- 在「热门内容」页**勾选**要同步的帖子（左侧复选框）
- 点右上「**同步选中到飞书**」→ 一键写入

## 关键概念

### 帖子状态（监控页表格里）

| 状态 | 含义 |
|------|------|
| 🟢 正常 | 上次抓取成功 |
| 🟠 🔒 需登录 | XHS 给该帖打了「登录可见」标签，匿名抓不到。可以打开 cookie 兜底，或忽略这条 |
| 🔴 已删除 | 帖子被作者删除 / 屏蔽 |
| 🔴 抓取异常 | 网络错误、HTML 解析失败等 |

### Cookie 状态（账号管理表格里）

| 状态 | 含义 |
|------|------|
| 🟢 正常 | cookie 有效 |
| 🔴 已失效 | cookie 过期，需要重新扫码登录 |
| ⚪ 未检测 | 还没跑过健康检查 |

每天 09:00 自动检查所有活跃账号，状态变化（正常 → 失效）会推送飞书。也可以手动点「**检查全部 Cookie**」。

### 观测帖子使用 Cookie 兜底（默认关闭）

XHS 现在对很多帖子加了登录墙，匿名访问会 302。这个开关打开后，**未绑定账号的观测帖子会借用任一有效账号的 cookie 来抓**。

**代价**：你的账号会被小红书后台记录「频繁访问别人的帖子」，长期可能触发风控。**建议专门弄个一次性账号承担此功能**。

## 常见问题

### Q1. 为什么有些帖子一直 🔒 「需登录」？

XHS 对**单条帖子**单独决定是否要求登录访问，依据可能是：作者隐私设置、内容含敏感词、被举报、或纯随机风控。这是 XHS 服务端策略，**客户端无法绕过**。

唯一可行的：开启「Cookie 兜底」用账号 cookie 抓，或者放弃这条帖子。

### Q2. xsec_token 会过期吗？需要重新分享获取新链接吗？

实测：**同一帖子的 xsec_token 是确定的**，重新分享拿到的 token 完全相同，**不存在「token 过期」**。被 302 是因为 XHS 给帖子加了登录墙，跟 token 无关。

### Q3. 为什么扫码登录的账号在小红书 App 里收到「异常登录」警告？

XHS 风控对「同账号在多设备并发登录 + 异常访问行为」敏感。建议：

- **专门用一个小号扫码**，不要用主力账号
- **不要同时在手机 App 和我们系统大量使用同一账号**
- **不要把同一账号扫码到多个第三方工具**

### Q4. trending 抓取每 30 分钟一次会不会被风控？

会。每次跑会用一个账号加载 4-N 个搜索页，几小时内累计上百次访问 → 触发风控。

**建议**：

- 把检测间隔从 30 分钟拉到 1-2 小时
- 关键词控制在 1-3 个
- 用专门的小号做 trending（即便被封也不影响主业）

### Q5. 飞书同步「成功 0 条，失败 N 条」怎么办？

先看真实错误。常见 4 种：

| 错误码 | 含义 | 解决 |
|--------|------|------|
| 99991672 | 没开通应用身份的 bitable:app | 飞书开放平台 → 权限管理 → 筛选「应用身份」开通 |
| 91403 | 应用没被加为表格协作者 | 多维表格分享 → 添加应用为可编辑协作者 |
| 91402 | 表格不存在 | 检查 app_token / table_id 是否填对 |
| 1254045 | 字段不存在 | 现在自动建字段，正常不应出现 |

### Q6. SOCKS5 代理为什么不能用？

Playwright 浏览器**不支持 SOCKS5 用户名密码认证**。如果你的代理服务商提供 SOCKS5+认证，请改用 HTTP/HTTPS 代理。无认证的 SOCKS5（IP 白名单）可用。

### Q7. 服务器内存只 1G+ 够用吗？

紧张。Playwright Chromium 单实例 ~500MB，加上 Python 进程 ~150MB，跑 trending 时容易 OOM。建议：

- 加 swap（`fallocate -l 2G /swapfile && mkswap /swapfile && swapon /swapfile`）
- 把 trending 间隔拉长，不让多个浏览器实例同时跑

## 运维常用命令

```bash
# 服务器（systemd）
systemctl status redbook        # 看状态
systemctl restart redbook       # 重启
journalctl -u redbook -f        # 跟日志

# 看应用日志
tail -f /var/log/redbook.log

# 数据库
sqlite3 database/monitor.db
# 重置某个 cookie 状态
sqlite3 database/monitor.db "UPDATE monitor_accounts SET cookie_status='unknown' WHERE id=N;"
# 重置 trending 改写状态
sqlite3 database/monitor.db "UPDATE trending_posts SET rewrite_status='pending' WHERE rewrite_status='failed';"
```

## 安全建议

1. **首次登录立刻修改 admin 密码**（目前需直接改 DB 里的哈希）
2. **JWT secret** 在 `database/.jwt_secret`，不要泄露
3. **如果暴露在公网**：建议加 HTTPS（nginx 反代或 uvicorn `--ssl-*`），不要让默认 admin 账号直接对外
4. **不要把同一 cookie 共享给多人/多设备**——风控会立刻盯上

## 项目限制

- **不抓私信**：XHS 私信接口风控严重，没有可靠方案
- **不抓粉丝/关注列表**：同上
- **不能搜不存在的内容**：只能抓 XHS 公开搜索结果里的帖子
- **AI 改写质量取决于你配的模型**：建议用 GPT-4o-mini 或 DeepSeek-v3 起步
